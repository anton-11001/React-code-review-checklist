# React Code Review Checklist

## 1. Загальна перевірка PR

* [ ] Код збирається і запускається без помилок.
* [ ] Немає runtime errors у браузері.
* [ ] Немає React warnings у console.
* [ ] Немає linter errors або warnings.
* [ ] PR вирішує саме ту задачу, для якої створений, без зайвих змін.
* [ ] Зміни не ламають існуючу функціональність.
* [ ] Перевірені основні user flows після змін.
* [ ] Є backward compatibility там, де це потрібно.
* [ ] Якщо змінений публічний API компонента, це явно враховано у всіх місцях використання.
* [ ] Якщо були змінені env, scripts, setup або інструкції запуску, README оновлений.
* [ ] У репозиторій не потрапили dist/build файли, IDE файли, секрети, локальні конфіги.
* [ ] `.gitignore` налаштований коректно.

---

## 2. Архітектура і загальний дизайн рішення

* [ ] Рішення відповідає існуючій архітектурі проєкту.
* [ ] Код синхронізований з існуючими code patterns і conventions у команді.
* [ ] Немає “випадкової” нової архітектури тільки в одному місці без причини.
* [ ] Логіка розкладена по правильних шарах:

  * [ ] UI
  * [ ] бізнес-логіка
  * [ ] API layer / services
  * [ ] helpers / utils
  * [ ] hooks
* [ ] Компоненти не знають зайвого про API або зовнішні джерела даних, якщо це можна винести.
* [ ] Повторювана логіка винесена в helper, hook, service або shared component.
* [ ] Дотримано SRP: один модуль або компонент має одну чітку відповідальність.
* [ ] Дотримано OCP: нова поведінка додається розширенням, а не постійною переробкою існуючого коду.
* [ ] Немає надмірної зв’язаності між модулями.
* [ ] Немає прихованих side effects у місцях, де їх не очікуєш.

---

## 3. Розмір і структура компонентів

* [ ] Компоненти невеликі та читабельні.
* [ ] Якщо компонент став занадто великим, він розбитий на child components.
* [ ] Орієнтир: якщо файл компонента перевищує приблизно **200–300 рядків**, варто оцінити доцільність декомпозиції.
* [ ] JSX не перетворився на величезне полотно.
* [ ] Орієнтир: великий суцільний JSX-block понад **50 рядків** варто винести у підкомпоненти або render helpers.
* [ ] У компонента немає занадто багатьох responsibilities одночасно:

  * [ ] fetch data
  * [ ] transform data
  * [ ] manage forms
  * [ ] render complex UI
  * [ ] handle modal logic
  * [ ] handle navigation
* [ ] Якщо компонент stateful без потреби, перевірити, чи можна спростити його.
* [ ] Stateless частини оформлені як окремі прості компоненти.
* [ ] Повторювані UI-блоки не копіюються вручну багато разів.

---

## 4. Якість JSX і рендер-логіки

* [ ] У JSX немає складної бізнес-логіки.
* [ ] У render не виконується зайва обробка великих даних на кожному рендері.
* [ ] Умовний рендеринг написаний читабельно.
* [ ] Немає глибоких вкладених ternary expressions, які важко читати.
* [ ] Складні умови винесені в змінні або helper functions.
* [ ] Мапінг списків виконаний коректно.
* [ ] Для елементів у масивах є стабільний і унікальний `key`.
* [ ] Не використовуються index keys там, де порядок може змінюватися.
* [ ] Немає створення нових важких об’єктів/масивів у JSX без необхідності.
* [ ] Не передається зайвий inline logic у children, якщо це ускладнює читання.
* [ ] Немає side effects у render.
* [ ] Не викликаються async operations прямо під час render.
* [ ] Не викликаються state updates під час render.

---

## 5. Props і component API

* [ ] Пропси компонента мінімальні й потрібні.
* [ ] Немає unused props.
* [ ] Немає props drilling без потреби.
* [ ] Імена props зрозумілі, передбачувані і консистентні.
* [ ] Boolean props мають коректні назви: `is...`, `has...`, `should...`, `can...`.
* [ ] Callback props названі як дії: `onClose`, `onSubmit`, `onChange`.
* [ ] Props не мутуються.
* [ ] Значення за замовчуванням задані там, де це потрібно.
* [ ] API компонента не перевантажений десятками прапорців, які важко підтримувати.
* [ ] Якщо компонент став занадто конфігурованим, перевірити чи не треба його розділити.
* [ ] Якщо змінено public props existing component, це не зламало інші місця використання.
* [ ] Нові props і параметри методів зберігають backward compatibility, якщо це важливо для проєкту.

---

## 6. Стан, data flow і React hooks

* [ ] State зберігається в правильному місці.
* [ ] Немає дублювання одного й того ж state у кількох місцях без причини.
* [ ] Не зберігається derived state, який можна обчислити.
* [ ] Немає direct mutation state.
* [ ] Оновлення state виконуються через setter/state management tools.
* [ ] Локальний state використовується там, де він справді локальний.
* [ ] Глобальний state не використовується для того, що можна тримати локально.
* [ ] Немає race conditions у state updates.
* [ ] Для залежного оновлення state використовується функціональний updater, де це потрібно.
* [ ] Hooks викликаються тільки на верхньому рівні компонента або custom hook.
* [ ] Не порушуються Rules of Hooks.
* [ ] `useEffect` використаний тільки для side effects, а не для того, що можна зробити під час render або через memoization.
* [ ] `useEffect` не створює нескінченні цикли.
* [ ] Dependency arrays коректні.
* [ ] Немає пропущених залежностей без чіткої причини.
* [ ] Якщо залежність свідомо пропущена, це обґрунтовано.
* [ ] `useCallback` використовується там, де це реально має сенс, а не механічно всюди.
* [ ] `useMemo` використовується для важких обчислень або стабілізації значень, а не бездумно.
* [ ] Якщо функція використовується в `useEffect`, перевірено чи потрібен `useCallback`.
* [ ] Custom hooks створені для повторюваної stateful logic.
* [ ] Custom hook має зрозумілий API і правильну назву з префіксом `use`.
* [ ] Cleanup в effects реалізований там, де потрібно.
* [ ] Таймери (`setInterval`, `setTimeout`) очищаються.
* [ ] Event listeners видаляються.
* [ ] Subscriptions / observers / sockets коректно відписуються.
* [ ] Немає API calling inside loops або effect loops через некоректні dependencies.

---

## 7. Асинхронність, API і data fetching

* [ ] API calls винесені в service layer / api layer.
* [ ] Компоненти не містять зайвий low-level код роботи з HTTP.
* [ ] Асинхронний код написаний консистентно: `async/await` або `Promise` chains, без хаотичного змішування.
* [ ] Усі API rejections обробляються.
* [ ] Є error handling для network failures, 4xx, 5xx, timeout cases.
* [ ] Немає “мовчазного” проковтування помилок.
* [ ] Error states правильно відображаються в UI.
* [ ] Loading states враховані.
* [ ] Empty states враховані.
* [ ] Retry / refetch логіка реалізована адекватно, якщо це потрібно.
* [ ] Немає дублювання однакових API викликів без потреби.
* [ ] Async methods оптимізовані:

  * [ ] незалежні запити виконуються паралельно, якщо це безпечно
  * [ ] послідовність використовується тільки там, де вона справді потрібна
* [ ] Немає memory leaks від незавершених async operations.
* [ ] Скасування запитів або ігнорування застарілих відповідей враховане, якщо це релевантно.
* [ ] Немає оновлення state після unmount.
* [ ] Мапінг response data до UI централізований і зрозумілий.
* [ ] Немає жорстко захардкоджених endpoint paths у компонентах.
* [ ] Немає дублювання error handling logic у багатьох місцях, якщо її можна централізувати.

---

## 8. TypeScript: типи, інтерфейси, generic usage

* [ ] Усі нові сутності мають коректні типи.
* [ ] Немає `any`, якщо без нього можна обійтись.
* [ ] Якщо `any` використано, це справді обґрунтовано.
* [ ] Інтерфейси та type aliases мають зрозумілі назви.
* [ ] Типи не дублюються без потреби.
* [ ] Спільні типи винесені в shared/types.
* [ ] Інтерфейси розширюються там, де це логічно, замість копіпасту.
* [ ] Типи для API response/request описані коректно.
* [ ] Props типізовані правильно.
* [ ] Return types очевидні або явно задані там, де це допомагає.
* [ ] Generic functions/components/hooks використані там, де вони реально підвищують перевикористання.
* [ ] Немає надмірно складних типів заради “красивості”.
* [ ] Типізація допомагає, а не ускладнює підтримку.
* [ ] Optional fields, nullable values і unions оброблені безпечно.
* [ ] Не ігноруються TypeScript errors через `as any` або unsafe assertions без причини.

---

## 9. Чистота коду і readability

* [ ] Код легко читати без додаткових пояснень.
* [ ] Імена змінних, функцій, компонентів і файлів змістовні.
* [ ] Назви відображають намір, а не технічну дрібницю.
* [ ] Function names описують дію.
* [ ] Boolean names описують стан.
* [ ] Немає абревіатур, які складно зрозуміти без контексту.
* [ ] Немає магічних чисел, рядків, шляхів, статусів або ключів.
* [ ] Hardcoded values винесені у constants/config/enums.
* [ ] Подібні значення згруповані в enum або constants object, коли це доречно.
* [ ] Код не дублюється.
* [ ] DRY дотриманий, але без overengineering.
* [ ] Складні шматки коду спрощені, якщо можна.
* [ ] Видалено мертвий код.
* [ ] Видалено unreachable code.
* [ ] Видалено commented-out code.
* [ ] Видалено debug artifacts.
* [ ] `console.log` відсутні.
* [ ] `console.error` / `console.warn` використовуються тільки якщо це частина усвідомленого logging strategy.
* [ ] Структура файлу логічна:

  * [ ] imports
  * [ ] constants
  * [ ] types
  * [ ] component/hook/function
  * [ ] exports
* [ ] Імпорти впорядковані та не містять невикористаного.
* [ ] Не імпортується зайве з бібліотек.
* [ ] Немає циклічних залежностей.

---

## 10. Коментарі та документація

* [ ] Коментарі пояснюють **чому**, а не дублюють **що** і так видно з коду.
* [ ] Немає зайвих або застарілих коментарів.
* [ ] Якщо в команді є правило документувати кожну функцію, це виконано.
* [ ] Якщо функція неочевидна, є короткий опис її призначення.
* [ ] Складні рішення, workaround-и або нетривіальні компроміси пояснені.
* [ ] README оновлений, якщо змінена поведінка або setup.
* [ ] Немає коментарів, які суперечать реальному коду.

> Примітка: правило “коментар над кожною функцією” краще робити як **team convention**, а не абсолютне правило. Для очевидних маленьких функцій це часто лише створює шум.

---

## 11. ES6+ і сучасний JavaScript стиль

* [ ] Використовується `const` / `let`, а не `var`.
* [ ] Використовується destructuring для objects/arrays, де це покращує читабельність.
* [ ] Використовується spread operator там, де це доречно.
* [ ] Arrow functions використовуються консистентно.
* [ ] Немає змішування старих і нових стилів без причини.
* [ ] Для transform operations використовується послідовний стиль.
* [ ] Не змішується безсистемно lodash, native JS і старі підходи для одних і тих самих задач.
* [ ] Код не ускладнений зайвими chaining/one-liners заради “красивості”.

---

## 12. Forms і user input

* [ ] Усі форми мають валідацію там, де це потрібно.
* [ ] Validation errors коректно показуються користувачу.
* [ ] Оброблені edge cases:

  * [ ] empty input
  * [ ] invalid format
  * [ ] too long input
  * [ ] server validation error
* [ ] Submit handler не допускає дубльованих сабмітів.
* [ ] Disabled / loading state для submit кнопки врахований.
* [ ] Поля мають коректні `label`, `name`, `id`.
* [ ] Форми доступні з клавіатури.
* [ ] Focus management після помилки або submit продуманий.
* [ ] Немає неконтрольованого змішування controlled/uncontrolled input без потреби.

---

## 13. Styling і UI consistency

* [ ] Стилі відповідають існуючій дизайн-системі або UI patterns.
* [ ] Перевага віддається спільним стилям/токенам/темі над локальним хаосом.
* [ ] Немає зайвого inline styling, якщо в проєкті прийнятий інший підхід.
* [ ] Немає копіпасту однакових стилів у багатьох місцях.
* [ ] Використовуються існуючі design tokens, spacing scale, colors, typography.
* [ ] Немає захардкоджених кольорів, відступів, z-index без потреби.
* [ ] Новий UI виглядає консистентно з іншими частинами застосунку.
* [ ] Hover / active / disabled / loading / error states враховані.
* [ ] Layout не ламається на типових розмірах екрана.
* [ ] Є перевірка responsiveness.
* [ ] Екран виглядає коректно на mobile/tablet/desktop, якщо це потрібно.
* [ ] Не використовується небезпечний або нестабільний CSS hack без пояснення.

---

## 14. Accessibility (a11y)

* [ ] Використовується semantic HTML там, де це можливо:

  * [ ] `button` замість clickable `div`
  * [ ] `nav`, `main`, `section`, `header`, `form`, `label`
* [ ] Інтерактивні елементи доступні з клавіатури.
* [ ] Є видимий focus state.
* [ ] Tab order логічний.
* [ ] `aria-*` атрибути використані тільки там, де semantic HTML недостатньо.
* [ ] Немає неправильного або зайвого використання ARIA.
* [ ] Усі form controls мають пов’язані labels.
* [ ] Для `label` використовується `htmlFor`.
* [ ] Помилки форми доступні для assistive technologies.
* [ ] Для динамічного контенту використовується `aria-live`, якщо це потрібно.
* [ ] Модалки, меню, dropdown-и, tabs правильно керують focus.
* [ ] Focus не губиться після route change або dynamic update.
* [ ] Зображення мають коректний `alt`.
* [ ] Якщо зображення декоративне, це явно позначено належним способом.
* [ ] Іконки без тексту мають accessible name, якщо вони інтерактивні.
* [ ] Контрастність тексту й елементів достатня.
* [ ] Немає залежності тільки від кольору для передачі змісту.
* [ ] Використання screen reader сценаріїв хоча б базово враховане.

---

## 15. Безпека

* [ ] Не використовується `dangerouslySetInnerHTML`, якщо це не абсолютно необхідно.
* [ ] Якщо HTML все ж рендериться, він санітизований.
* [ ] Не зберігаються чутливі дані у небезпечних місцях без причини.
* [ ] Немає витоку токенів, секретів, внутрішніх URL або credentials.
* [ ] User input не вставляється безпечним чином у DOM, URL або query params.
* [ ] Обробка помилок не розкриває зайву внутрішню інформацію.
* [ ] Враховані basic XSS-related ризики.
* [ ] Посилання на зовнішні ресурси відкриваються безпечно, якщо це потрібно.
* [ ] Немає небезпечних залежностей, доданих без перевірки.

---

## 16. Продуктивність

* [ ] Немає зайвих ререндерів.
* [ ] Перевірено, чи не передаються нові object/function references без потреби.
* [ ] Великі списки оптимізовані там, де це потрібно.
* [ ] Heavy calculations не виконуються на кожен render без потреби.
* [ ] Memoization використовується обґрунтовано.
* [ ] Не відбувається дублювання fetch-ів.
* [ ] Немає nested expensive computations у JSX.
* [ ] Імпорти зроблені акуратно, без зайвого тягнення великих шматків бібліотек.
* [ ] Перевірено нові бібліотеки на розмір bundle.
* [ ] Якщо додано нову library, перевірено:

  * [ ] чи вона справді потрібна
  * [ ] чи немає вже такого ж рішення в проєкті
  * [ ] чи немає легшої альтернативи
  * [ ] чи активна підтримка бібліотеки
* [ ] Немає дублювання функціональності вже існуючими npm packages.
* [ ] Видалені невикористані npm packages.
* [ ] Lazy loading / code splitting використані, якщо це доречно.
* [ ] Зображення, ресурси та важкі UI pieces завантажуються розумно.

---

## 17. Redux / state management, якщо використовується

* [ ] Reducers не мутують state, якщо не використовується дозволений abstraction layer типу Redux Toolkit + Immer.
* [ ] Store містить тільки те, що справді має бути глобальним.
* [ ] Selectors використовуються там, де це допомагає.
* [ ] Немає дубльованих derived values у store.
* [ ] Async logic розміщена у відповідному місці.
* [ ] Немає зайвого coupling між UI і store internals.
* [ ] State shape читабельний і підтримуваний.
* [ ] Actions / reducers / slices мають зрозумілі назви.

---

## 18. Тести

* [ ] Є unit tests для критичних частин.
* [ ] Є тести для складної бізнес-логіки.
* [ ] Є тести для edge cases.
* [ ] Є перевірка ключових user interactions.
* [ ] Тести не зав’язані на implementation details без потреби.
* [ ] Новий функціонал не додається без тестів, якщо в проєкті це є стандартом.
* [ ] Існуючі тести не зламані.
* [ ] Не додані flaky tests.
* [ ] Mock-и не приховують реальні проблеми.

# Важливі уточнення до деяких пунктів

Ось кілька пунктів, які краще трактувати не буквально, а розумно:

**“Кожна функція повинна мати коментар зверху”**
Краще так: коментарі потрібні для неочевидних рішень і складної логіки. Для дрібних очевидних функцій це часто зайве.

**“JSX не більше 50 рядків”**
Це хороший орієнтир, але не абсолютне правило. Головне — читабельність і декомпозиція.

**“Використовувати useCallback з useEffect”**
Не завжди. `useCallback` потрібен не тому, що так “треба”, а коли важлива стабільність посилання або це реально допомагає уникнути зайвих перерахунків/ефектів.

**“Стилі в компонентах гірші за common styling file”**
Це залежить від підходу в проєкті: CSS modules, styled-components, MUI sx, Tailwind, theme tokens. Тут важливіше консистентність з існуючим патерном.

**“Create custom hooks”**
Не треба робити custom hook заради самого hook. Виносити варто тоді, коли логіка повторюється або відділення реально покращує читабельність.
